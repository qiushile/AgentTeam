# 业主投票管理系统 - 项目规划文档

> 编制人：项目交付经理 | 日期：2026-05-06 | 来源：产品部功能清单（2026-03-26）

---

## 一、项目概述

**系统定位**：为街道/社区、业委会和业主提供合规、透明的线上业主大会投票平台。

**三端架构**：
1. **街道/社区管理端** — 审核与监管
2. **业委会管理端** — 投票发起与管理
3. **业主小程序端** — 认证、参会、投票

## 五、风险矩阵

| 风险 | 概率 | 影响 | 缓解策略 |
|---|---|---|---|
| 票权规则配置过于灵活导致复杂度失控 | 高 | 高 | MVP 先固定 2-3 种常用规则，后续迭代扩展 |
| 面积与人数双维度统计逻辑混淆 | 中 | 高 | 设计阶段明确抽象层，编写单元测试覆盖 |
| 微信审核周期不可控 | 中 | 中 | 提前准备小程序提审，预留 1-2 周 buffer |
| 100+ 并发性能不达标 | 低 | 高 | 架构设计时引入 Redis 缓存，提前压测 |
| 业主清册数据质量差（导入失败率高） | 中 | 中 | 提供数据校验工具 + 导入模板 |
| 法定投票规则变更 | 低 | 高 | 规则引擎独立模块，配置化管理 |

## 七、非功能性需求落地

| 类别 | 要求 | 落地方案 |
|---|---|---|
| 性能 | 1000 户，100+ 并发 | Redis 缓存 + 读写分离 + 限流 |
| 安全 | 数据加密、权限、日志 | HTTPS + AES 敏感字段 + RBAC + 审计日志 |
| 可用性 | 7×24 小时 | K8s 多副本 + 健康检查 + 自动重启 |
| 扩展 | 微服务架构 | Spring Cloud / 服务拆分 + API Gateway |
