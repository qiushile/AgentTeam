| **service-store** | 商户管理 | ✅ 已实现 |

## 二、与业主投票系统需求的差距分析

### 🔴 2.1 核心缺失 — 投票系统

| 缺失项 | 现状 | 需要补充 | 优先级 |
|--------|------|----------|--------|
| **投票主表** | 无 | vote 表（标题/类型/状态/时间范围） | P0 |
| **投票选项表** | 无 | vote_item 表（选项内容/附件） | P0 |
| **投票记录表** | 无 | vote_record 表（投票人/选项/权重/时间） | P0 |
| **票权规则引擎** | 无 | vote_rule 表 + 策略模式引擎 | P0 |
| **审核流程状态机** | 无 | 投票审核流程（草稿→待审核→通过/拒绝→进行中→已结束） | P0 |
| **统计快照表** | 无 | vote_stat_snapshot 表（定时统计） | P1 |
| **微信小程序投票端** | 无 | 业主投票小程序页面 | P0 |
| **投票管理后台** | 无 | Vue 管理后台投票模块 | P0 |

### 🔴 2.2 现有 questionnaire 模块的不足

MicroCommunity 已有 `question` 模块（问卷/调查），但 **不满足投票需求**：

| 维度 | Question 模块（已有） | 投票系统（需要） |
|-----|----------------------|----------------|
| **用途** | 满意度调查、意见收集 | 法定表决（业主大会决议） |
| **票权计算** | 一人一票，无权重 | 按面积/户数/双过半计算 |
| **法律依据** | 无 | 《民法典》第 278 条双过半校验 |
| **审核流程** | 无审核，直接发布 | 多级审核（业委会→管理员） |
| **匿名性** | 实名回答 | 结果展示匿名 |
| **不可篡改** | 可修改 | 投票后不可更改 |
| **状态管理** | 无状态机 | 草稿→审核→进行中→已结束 |
| **防重复** | 无 | 分布式锁 + 唯一索引 |
| **法律效力** | 无 | 需要符合法定程序 |

**结论：不能复用 question 模块，需要全新开发投票系统。**

### 🟡 2.3 现有 owner 模块的不足

| 缺失项 | 现状 | 需要补充 |
|--------|------|---------|
| **业主身份核验** | OwnerDto 有基本信息 | 增加房产证验证、身份证 SHA256 脱敏存储 |
| **房屋-业主关联** | OwnerRoomRelDto 存在 | 需完善面积关联，用于票权计算 |
| **业主投票资格** | 无 | 需增加投票资格校验逻辑 |
| **业主认证审核** | 无 | 需增加业主认证审核流程 |

### 🟡 2.4 现有 ownerCommittee 模块的不足

| 缺失项 | 现状 | 需要补充 |
|--------|------|---------|
| **业委会权限** | OwnerCommitteeDto 有基本信息 | 需关联投票创建权限 |
| **业委会角色权限** | 无 | 需在权限系统中增加业委会角色 |

### 🟡 2.5 现有 privilege 权限体系的不足

| 缺失项 | 现状 | 需要补充 |
|--------|------|---------|
| **投票专用权限** | 有基础 RBAC | 需新增投票创建/审核/查看权限点 |
| **投票数据隔离** | 有 dataPrivilege | 需确保投票数据按小区隔离 |

## 六、建议

1. **不要复用 question 模块** — 问卷和投票在法律性质、票权计算、审核流程上完全不同，强行复用会导致代码复杂且难以维护
2. **新建 service-vote 微服务** — 投票是独立业务域，应独立部署
3. **优先实现双过半策略** — 这是《民法典》第 278 条的法定要求，也是业主投票的核心
4. **投票记录不可篡改** — 这是法律效力的保障，需要 write-once 设计
5. **充分利用现有数据** — 小区/房产/业主/权限等基础数据可直接复用
